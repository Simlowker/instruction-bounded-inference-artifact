# Why we did NOT use Qwen 2.5 1.5B Q6_K (the originally planned target):
#   The 1.46 GB Q6_K GGUF triggered IC0524 ("Exceeded the limit for the
#   number of accessed pages in the stable memory in a single message
#   execution: limit 2_097_152 KB for regular messages") on load_model.
#   This is a hard ICP runtime limit, NOT our bug, and is itself a
#   useful finding for the paper: Q6_K models above ~1 GB are not
#   viable on current ICP without splitting the load across messages.

# → Bit-exact equivalence between scalar and vectorized Q6_K unpack

#   bit-exactness check (matmul-bench, all 8 reps result=56754044928).
#
# E2E observation:
#   The microbench reports +2.3% (357.81 → 349.85 ns/iter). At
#   N_MAX = 25, +2.3% buys 0.575 tokens — invisible to the integer
#   tok/call metric. To resolve a +2.3% gain on this metric, the
#   model would need to produce ~50 tok/call (≥ 1.15 token delta) or
#   the metric would need to be cycles-per-token rather than
#   tokens-per-call. Neither is achievable for this model size on
#   the current ICP runtime (cycles-per-call is not exposed via the
#   standard `dfx canister call` response).
#
# What C4 still contributes (independent of the E2E null result here):
#   1. Microbench-level +2.3% kernel speedup, reproducible (CV 2.5%
#      patched vs 8.0% baseline → 3.2× variance reduction).
#   2. Bit-exact preservation across the kernel rewrite (verified at
#      both microbench and on-chain).
#   3. Strict-SIMD only (no relaxed_simd ops) → preserves determinism
#      under deterministic-WASM constraints (e.g., ICP).
#   4. Upstream-PR-able patch (small, isolated, microbench-backed).
